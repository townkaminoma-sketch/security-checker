## 3. 固定原則

### 3.1 開発原則

- 人間承認つきで段階的に進める
- 1 Issue = 1 checkpoint
- 1 PR = 1 目的
- 最小差分で実装する
- スコープ外の変更は禁止
- 大規模リファクタは禁止
- テスト未実行での採用は禁止
- CI が落ちたら merge しない
- 自動 merge は行わない

## 4. 変更禁止範囲

以下は原則として変更禁止です。

- 認証情報 / 秘密情報 / 環境変数の扱い
- 既存の checkpoint と無関係なファイル
- 大規模なディレクトリ再編

変更が必要だと判明した場合は、その場で広げず、次 checkpoint 候補として分離すること。

## 6. Done 条件

今回の checkpoint は、以下をすべて満たしたときのみ Done とみなします。

1. スコープ外変更がない
2. 実装差分が最小に保たれている
3. テストが実行され、結果が記録されている
4. lint / CI が通っている
5. PR に目的・非目的・変更内容・影響範囲・テスト結果・rollback 手順が記載されている
6. 人間が次の checkpoint に進める状態と判断できる

---

## 7. Rollback 条件

以下のいずれかに該当した場合、rollback 候補とします。

- スコープ外変更が含まれている
- テスト未実行または失敗状態である
- CI が失敗している
- 元に戻しにくい変更が含まれている

## 8. AI 自己チェック項目

### 実装前
- 今回の目的は明確か
- 非目的を侵食していないか
- 変更対象ファイルは限定されているか
- 今回は 1 checkpoint だけか
- rollback しやすい差分か

### 実装後
- 目的を満たしているか
- スコープ外変更がないか
- テストは実行したか
- PR に記録すべき内容を整理したか
